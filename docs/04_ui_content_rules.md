# Story Archive UI 内容规则

这些规则用于后续重写 `js/data.js` 和 Story Archive UI。当前阶段不改 UI 文件。

## 1. 故事卡片应该显示哪些字段

每张故事卡片建议显示：

- `title`: 故事标题，例如“先锋四巷门楼”。
- `category`: 类型，例如历史建筑、历史环境要素、非遗美食、社区生活。
- `placeName`: 点位名称。
- `address`: 具体地址；没有地址时显示“资料不足，需人工补充”。
- `period`: 年代或时期；不能确定时不要硬写。
- `sourceType`: 官方规划资料、社区展陈资料、历史信息整理、居民口述模拟、学校项目采集。
- `sourceNote`: 来源文件和页码。
- `archiveDetail`: 1-2 句档案依据。
- `tags`: 关键词。
- `credibility`: high、medium、low、conflict。
- `suggestedImageStyle`: 用于 CSS 封面生成的风格 token。
- `availableMedia`: 照片、历史介绍、相关记忆、音频、任务；没有真实媒体时明确显示占位状态。

## 2. 故事详情页应该分成哪些区块

详情页建议固定分区：

1. 标题区：标题、类别、年代、地址、可信度。
2. 来源区：sourceType、sourceFile、sourcePage、sourceNote。
3. 历史档案：只写资料里能证明的事实。
4. 空间细节：墙体、门楼、麻石、古井、街巷关系等可观察内容。
5. 社区口述占位：没有真实采集时只能写“待采集问题”，不能冒充居民原话。
6. 参观提示：visitorPrompt，用问题引导用户观察。
7. 儿童任务：如找材料、认巷道、画鱼骨地图、问长者。
8. 资料缺口：缺页、扫描页、地址冲突、待 OCR、待现场核对。
9. 上传入口：允许用户围绕该点位提交文字、照片、音频、视频和地理位置，提交后进入管理员审核。

## 3. 没有真实图片时，如何用 CSS imageStyle 生成封面视觉

不要用假照片。可以用 CSS 生成抽象封面，但要和内容有关：

- `shell-wall-texture`: 蚝壳墙，使用浅灰、米白、贝壳椭圆纹理。
- `bluebrick-oyster-gable`: 青砖 + 蚝壳墙 + 镬耳山墙，使用青灰砖格和曲线山墙轮廓。
- `old-well-marker`: 古井，使用圆形井口、石纹、深色中心。
- `granite-paving`: 麻石巷，使用不规则石板格。
- `brick-gatehouse`: 门楼，使用青砖门洞、麻石门框。
- `redbrick-lane`: 前锋十八巷，使用红砖、窄巷透视、阳台线条。
- `fishbone-map`: 鱼骨状街巷，使用一条主线和多条支线。
- `schoolyard-memory`: 先锋小学，使用操场线、教室窗格、旧校门轮廓。
- `water-festival-lantern`: 水色，使用水波和灯彩形。
- `missing-recipe-card`: 资料不足的美食，使用问号、食谱纸、空白卡。

封面必须标注“CSS visual / no real photo”或用小字说明“示意视觉，不是真实照片”。

## 3.1 地图与时间轴内容规则

- 地图方向：第一版采用手绘地图风格，不少于 10 个地点节点。来源：`100个问题和答案.docx`，第 115 段。
- 必须优先出现的地点：先锋大街、前锋大街、新兴大街、塘边大街、前锋十八巷、大东路、大西路、光明南路、大北路、前锋十二巷、先锋四巷、前锋五巷、前锋七巷、大东路三巷、大东路一巷、原先锋小学、工会建筑群。来源：`100个问题和答案.docx`，第 116 段。
- 每个地点节点尽量包含：照片、历史介绍、相关记忆、音频、任务。没有真实内容时显示“待采集”。
- 时间轴分段：北宋、明清、民国、建国初期、改革开放、当代。来源：`100个问题和答案.docx`，第 115 段。
- 主题线：水陆枢纽、市桥墟市、先锋小学、非遗美食、轮滑、宗祠/庙祠、老街巷、社区生活。来源：`100个问题和答案.docx`，第 116 段。

## 4. 哪些内容必须显示 sourceNote

以下内容必须显示来源：

- 任何具体地址。
- 任何具体数量，例如 7 处蚝壳墙、3 处门楼、5 处麻石巷、2 处古井、27 条传统街巷。
- 任何保护等级，例如不可移动文物、历史建筑、传统风貌建筑线索。
- 任何年代判断，例如清代、清末、民国 1931 年后、1949 年、2000 年。
- 任何规划定位，例如街区博物馆、广府文化体验街区、老城市新活力。
- 任何非遗名录、传统技艺、代表性项目。
- 任何资料不足或资料冲突提示。

## 5. 哪些信息不能胡编

不能编：

- 北宋石板桥的具体位置、桥名、故事和人物。
- 蚝壳墙的建造者、家族、传说。
- 古井井名、井龄、用途故事。
- 居民原话、长者回忆、学校师生故事。
- 非遗传承人的活动细节。
- 美食做法、店铺历史、老字号名称。
- 工商银行旧址、姑婆屋的具体地址和单体故事。
- 任何未在资料中出现的人名、年代、事件。

写法规则：如果资料没有明确细节，就写“资料不足，需人工补充”，并把它转成采集问题。

## 6. 如何区分来源类型

### 官方规划资料

包括：

- `1先锋大街前锋大街保护发展规划(文本）.pdf`
- `2先锋大街前锋大街保护发展规划(说明书）.pdf`
- `4先锋大街前锋大街保护发展规划（宣传册）.pdf`
- `市桥复心计划2025.pdf`

用法：可作为保护对象、数量、规划定位、空间结构、非遗名录的主要依据。

### 社区展陈资料

包括：

- `拾光源点•先锋社区 1.0.pdf`
- `先锋社区智能体项目提案.pdf`

用法：可作为项目定位、客群、运营空间、演示目标和体验语气依据。不能单独扩写历史事实。

### 历史信息整理

包括：

- `市桥先锋大街历史信息搜集.pdf`
- `市桥先锋大街历史信息搜集ChatGPT.pdf`

用法：可作为补充线索，使用时要低一档可信度；如内容与规划资料冲突，以官方规划资料优先，并标记冲突。

### 居民口述模拟

当前没有真实口述采集。UI 里只能显示：

- “待采集问题”
- “可能采访对象”
- “口述占位，不代表真实居民原话”

不能写成“阿婆说”“老街坊回忆”等真实口吻。

### 学校项目采集

当前没有学校采集成果。可以设计任务：

- 学生拍摄古井和麻石巷。
- 学生采访长者，但必须有授权。
- 学生绘制鱼骨街巷地图。

成果上线前必须有采集日期、采集人、审核状态和授权说明。

### 需求问答资料

包括：

- `100个问题和答案.docx`

用法：只能作为项目目标、MVP 范围、用户角色、页面结构、数据字段、交互方式、审核规则的依据。不能作为历史事实依据。

## 6.1 上传、审核和隐私规则

根据 `100个问题和答案.docx` 的已填答案：

- 上传内容类型：文字、照片、音频、视频、地理位置。
- 最小字段：标题、讲述人、时间、地点、故事正文。
- 内容类型要区分：个人记忆、家庭记忆、社区事件、建筑记忆、美食记忆、学校记忆。
- 讲述人可记录年龄段，不强制精确年龄。
- 需要隐私保护、授权同意和公开范围选择。
- 公开范围：公开、仅社区可见、仅管理员可见、不公开仅存档。
- 上传内容必须经过管理员审核。
- 管理后台第一版功能：查看提交、编辑、删除、审核、加标签、导出数据。
- 对涉及真实人物、家庭、住址、学校儿童照片的内容，默认不要公开，必须经过授权和审核。

## 7. 如何写成“展厅故事牌 + 社区口述 + 历史档案”的混合风格

建议每条故事采用三层写法：

### 展厅故事牌

短、清楚、适合站着看。示例结构：

- “这里是什么”
- “为什么重要”
- “你可以看哪里”

### 历史档案

严谨、带来源。示例结构：

- 来源文件
- 页码
- 原始事实
- 可信度

### 社区口述

当前只能写成“待采集问题”，例如：

- “你小时候是否从这条巷经过？”
- “这口井以前还有人用吗？”
- “前锋十八巷的红砖楼以前住过哪些人？”

等真实采集完成后，再把口述内容挂入故事详情，并保留采集人、时间、授权和审核状态。
